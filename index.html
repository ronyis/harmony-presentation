Messages arrive via SQS, so any pod can pick up any message — there's no sharding or affinity. Each pod maintains an in-memory registry mapping thread_id to a live agent instance. If the pod has already handled this thread, it reuses the cached agent. Otherwise, it creates a new one and loads messages and state from Postgres. The database is the source of truth — the registry is just an optimization.

The engine is built on four pillars. Toolkits for modularity, gates for conditional activation, dependencies for lazy injection, and persistent state. Let's look at each.

Toolkits

"The Armory"

Think of toolkits as the weapons in a knight's armory. Each one gives the agent a specific capability.

What is a Toolkit?

A self-contained bundle of everything the agent needs for one capability

🔧 Tools

Actions the agent can call (API calls, queries, mutations)

📝 Instructions

Dynamic prompts injected into the agent's context

📦 Dependencies

Services & clients needed by the tools

🚪 Gate

Condition that enables or disables the whole toolkit

A toolkit has 4 parts. Tools are the actual functions the LLM can call. Instructions are dynamic prompts. Dependencies are the services those tools need. And a gate decides if the whole toolkit should be active for this specific run.

Anatomy: Service Catalog Toolkit

SERVICE_CATALOG_TOOLKIT = Toolkit(
  name="service_catalog",

  # 🔧 Tools the agent can call
  tools=[
    Tool(evaluate_form_progress, max_retries=3),
    Tool(create_ticket_from_catalog, max_retries=4),
  ],

  # 📦 Lazy-loaded dependencies
  required_deps={
    "service_catalog_service": lambda deps: ServiceCatalogService(...),
    "tickets_service": lambda deps: TicketsService(...),
  },

  # 📝 Dynamic instructions
  instructions=[build_catalog_prompt],

  # 🚪 Only active if tenant has catalog items
  gate=has_catalog_items,
)
Here's a real example — the service catalog toolkit. It has two tools for form evaluation and ticket creation, lazy deps for the services it needs, a dynamic prompt builder, and a gate that only activates it if the tenant has catalog items.

Gates

"Who Shall Pass?"

Gates are the gatekeepers. They decide which toolkits are active for any given run.

What are Gates?

Async functions that decide if a toolkit should be active or hidden

Gate returns True

  • Toolkit's tools are visible to the LLM
  • Toolkit's instructions are injected
  • Dependencies are available

Gate returns False

  • Toolkit completely hidden
  • No instructions added
  • Dependencies never built (saves resources)
A gate is an async function that runs at the start of each agent step. If it returns true, the toolkit is fully active. If false, the toolkit is completely invisible — tools hidden, instructions not injected, dependencies not even created.

Gate Examples

# Only show service catalog if tenant has items
has_catalog_items(ctx):
  categories = await service.list_categories()
  return any(c.items for c in categories)
# Only show password reset if user has Okta or Entra
requires_supported_idp({SupportedIdp.OKTA})(ctx):
  for source in employee.sources:
    if source.name in supported_idps:
      return True
  return False
Two real examples. First: the catalog gate checks if the tenant even has catalog items — no point showing catalog tools if there's nothing to browse. Second: the IDP gate checks if the user's identity provider supports the operation — no Okta password reset if the user isn't on Okta.

Tool-Level Gates: prepare Callbacks

Fine-grained control — hide individual tools within a toolkit

# Hide "create ticket" if a ticket already exists
hide_tool_if_ticket_exists(ctx, tool_def):
  if ctx.deps.state.ticket_id:
    return None  # Tool hidden from LLM
  return tool_def  # Tool visible
Beyond toolkit-level gates, we have tool-level prepare callbacks. This example hides the "create ticket" tool once a ticket already exists in the conversation. The LLM literally cannot see or call the tool. Prevents duplicates.

Dependencies

"Everything You Need, When You Need It"

Dependencies — our dependency injection system. Like a squire's pack: you only unpack what you need for the current fight.

Lazy Dependency Injection

Dependencies are built on first access, not upfront

Eager (wasteful)

Build all 15+ services at startup

Even if gates disable 10 of them

Lazy (efficient)

Build a service only when its tool is actually called

Gate disables toolkit? Deps never built

Why lazy? Because with 15 toolkits, building all dependencies upfront is wasteful. If the IDP gate disables password reset, why build the IDP client? Lazy deps mean we only pay for what we use.

How Dependencies Work

Here's the flow. Each toolkit declares what deps it needs. The DependencyRegistry collects all of them and dynamically creates an AgentDeps subclass with LazyDep descriptors. When a tool accesses ctx.deps.some_service, it's built on the fly and cached.

Skills — Dynamic Toolkits

A Skill is a Toolkit that starts dormant and is activated mid-conversation

Skills Catalog
shows available descriptions
→
load_skill
activates silently
→
Tools Unlocked
tools & instructions active
  • The LLM decides when to load a skill based on user intent
  • Loaded skills persist in state across runs
  • Avoiding context bloat
Skills are dynamic toolkits. They're registered with the agent but start dormant — tools hidden, instructions not injected. A dynamic catalog is injected into the prompt listing available skills. When the user's request matches a skill, the LLM calls load_skill silently. From that point on, the skill's tools and instructions are active. This keeps the LLM's context lean — only the capabilities it needs are loaded. Loaded skills persist in state, so they survive across conversation turns.

Smart Filtering

SNS message attributes enable subscription-level filtering

// Message attributes published with every message
{
  "role": "employee | system | agent",
  "tenant_id": "uuid",
  "thread_source": "slack | teams | portal",
  "is_internal": "true | false",
  "user_source": "slack | teams | portal",
  "system_source": "agent | service-desk",
  "assigned_agent": "helpdesk-agent | workflow-builder"
}

Each subscriber only gets messages it cares about.
e.g., Service Desk processor filters out messages from itself — prevents loops.

Each message carries attributes. Subscribers use these for filtering at the SNS level. The service desk processor filters out messages that originated from itself — preventing infinite loops. Slack processor only gets Slack-source threads.

Why FIFO?

Ordering

Messages within a thread are always processed in order.

Grouped by thread_id — no cross-thread blocking.

Deduplication

Each message processed exactly once.

Message ID used as deduplication key.

Conversations make no sense out of order.

We use FIFO for two reasons. First: ordering. Messages in a conversation must be processed in sequence. FIFO guarantees this within a message group (thread_id). Second: deduplication. No double-processing.
